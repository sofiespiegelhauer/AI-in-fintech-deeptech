🎯 Precision vs Recall

THE rule: find the disaster

Ask one question: which mistake is the disaster?

Miss (FN) = disaster → chase RECALL — cast a wide net, tolerate noise.
False alarm (FP) = the expensive one → chase PRECISION — only act when sure.

Never "always recall" — find the disaster every single time. (Wildfire alert → recall. Small-business spam filter → precision, because the lost customer email IS the disaster.)

Instant classifiers

Lives/safety/security at stake if missed → recall (cancer screening, fire alarm, overheat shutdown, passport impostor).
Blocking/rejecting real people or losing real opportunities → precision (spam filter, content takedowns, auto-rejecting candidates).

Capacity override: if the review team can only handle 40 of your 200 daily flags → precision first, whatever the stakes. Recall you can't act on is worthless.

When to tolerate FNs vs FPs (the seesaw)

You never want errors — you choose which to tolerate. Lower threshold → more FPs, fewer FNs (= recall). Raise threshold → more FNs, fewer FPs (= precision).

Tolerate FPs when a miss is catastrophic/irreversible → chase recall (cancer, wildfire, impostor, big-loss fraud).
Tolerate FNs when the false alarm is the harmful one → chase precision (spam filter, auto-rejecting candidates, blocking real customers, maxed review capacity).

Hooks: "FN is silent, FP is noisy" — if the silent, later, bigger cost is unsurvivable, accept the noise. And ask who suffers each error: FP landing on innocent people at scale → precision; FN landing on a victim → recall.

Formula picker

"Of my flags, how many right?" → precision = TP/(TP+FP) (column).
"Of the real cases, how many caught?" → recall = TP/(TP+FN) (row).
"One number, imbalanced classes, both errors matter" → F1 = 2PR/(P+R).
"How often right overall?" → accuracy — distrust it whenever one class is rare (ask the base rate first).

🩺 Diagnosis rules (what went wrong?)

First move, always

Suspect the data before the algorithm (80/20). Then ask: data problem or deployment problem?

Bad from day one → data/labels or overfitting. Good at launch, decayed later → drift/world changed. Fine in the lab, failed in rollout → deployment/verification/monitoring.

Overfitting or distribution shift?

Failed on new data from the same world → overfitting (memorised, didn't generalise).
Worked fine, then the world changed → distribution shift.

Bonus split: inputs changed → data drift; input→outcome relationship changed → concept drift.

Which failure shape? (3-way)

KPI looks great while reality gets worse → Metric Corruption.
Incentives/silos/leadership caused it → Organisational Failure.
World moved on, or the system changed the behaviour it predicts → Adaptive Failure.

Which bias? (4-way, in order)

① Is the label itself a past biased decision? → historical/label bias.
② Sensitive attribute removed but correlated features rebuild it? → proxy bias.
③ Do the model's own outputs feed future training data? → feedback loop bias.
④ None of the above, group just missing from the data? → representation bias.

"There's bias" is not a diagnosis — name the mechanism; the cure differs each time.

Which proxy creep? (3-way)

"We did it" = "it works" → activity proxy (verify, don't just deploy).
"It answered" = "it's correct" → output proxy (ground + validate against source).
"Our own numbers say it's great" → self-reported proxy (demand independent audit).

Quick tell: who verified, and were they independent of who built/claimed it?

Hallucination triggers

LLM asked for facts, numbers, citations, current events, or policy without retrieval → assume hallucination risk. Plausible ≠ correct.

Fix by cause: stale knowledge → RAG; wrong facts → validation gate vs source; style/domain → fine-tune; high stakes → human check. LLMs are for drafting/summarising/exploring — never for retrieval or arithmetic.

🤖 AI/ML quick tests

Which AI family is this?

Reads/classifies/extracts existing data → cognitive/NLP.
Creates new content from a prompt → generative.
Takes multi-step autonomous action on real systems → agentic.

Misclassify → wrong architecture and wrong risks. The tell for agentic: it does things (refunds, bookings, emails) without per-step approval.

Which learning type?

Have labelled examples with known outcomes → supervised.
No labels, want structure/groups → unsupervised.
Learns from rewards/penalties of its own actions → reinforcement.

The learning type determines what data you need — most business AI is supervised on historical outcomes.

Should an LLM do this task?

Drafting, summarising, translating, explaining, brainstorming → yes (with human review).
Exact arithmetic, current facts, citations, guaranteed accuracy, final regulated decisions → no — or only with RAG + validation gate + human sign-off.

Rule: use an LLM to think, draft and explore — never to retrieve facts or calculate.

RAG or fine-tune?

Facts that change (policies, prices, regulations) → RAG.
Tone, style, domain behaviour → fine-tune.
Serious enterprise deployment → usually both: fine-tune for behaviour, RAG for facts.

How much autonomy for an agent?

Autonomy is a dial, not a switch. Compute (per-step)^steps first. Human gate wherever a step is irreversible, costly, or regulated. Keep chains short; checkpoint between steps; kill switch at a defined loss level.

0.95¹⁰ ≈ 60% — ten unsupervised steps is a coin flip with style.

Trust a vendor accuracy claim?

Ask in order: ① what's the base rate? (99% accuracy on 0.2% fraud = useless) ② precision AND recall, by subgroup? ③ who measured it — independent or self-reported? ④ tested on data from our world, or theirs?

"99.1% accuracy, validated internally" contains three red flags in five words.

🏛 Governance quick tests

The GDPR 5-second scan

Data collected for X, used for Y → purpose limitation (new purpose = new legal basis).
"Delete my data" + trained model or blockchain → erasure collision.
Automated decision about a person, no explanation/appeal → Art. 22 / explainability.
Consent from years ago stretched to a new use → consent decay.
Collected more than the purpose needs → data minimisation.

Is it EU AI Act high-risk?

Does the system decide about people's lives or livelihoods? Hiring, credit scoring, healthcare, justice/enforcement → high-risk → conformity assessment, documented lineage, human oversight, explainability — before deployment.

Rule: high-risk turns a modelling choice (like a threshold) into a legal one.

Who's accountable?

Business owns → business is accountable. IT manages, stewards monitor. Controller decides why/how (accountable); processor just executes.

If no named owner exists → that IS the governance failure (warning sign #1).

Is the human oversight real?

Check the override rate. If the human reviews everything but overrides ~nothing → performative oversight / automation bias, not a safeguard.

Place the human gate where the action is irreversible, costly, or regulated.

⚛ Quantum quick tests

Which concept is being described?

Many states at once until measured → superposition.
Two qubits, measure one, other instantly fixed → entanglement.
Wrong answers cancel, right one amplifies → interference.
Qubit ruined by heat/vibration → decoherence.

Which algorithm?

Searching faster → Grover (quadratic, √N).
Breaking RSA/ECC/ECDSA by factoring → Shor (exponential).

Encryption mentioned at all → it's Shor.

Do I need to act on quantum NOW?

Secrecy lifetime test: must the data stay secret longer than ~10 years? Yes → HNDL applies, start PQC migration now (migration itself takes 5–10 yrs). No → watch and track, don't bet.

Deadlines to cite: NIST deprecates RSA 2030, prohibits 2035; NSA 2027.

Quantum hype detector

Ask for the triple: which hardware, which algorithm, which problem shape? Can't name all three specifically → hype. Only 4 shapes benefit: optimisation, simulation, factoring, search.

Extra red flags: "room temperature", "millions of qubits", "no error correction needed" — all contradict NISQ reality.

⛓ Blockchain quick tests

Do you even need a blockchain?

Do independent, mutually-distrusting parties need shared truth with no trusted middleman?
Yes → blockchain has a case. No (same company, existing trusted party) → use a database (necessity vs overhead).

Public or private chain?

Need KYC/identity, privacy from competitors, speed, regulator comfort → private/permissioned (that's why banks choose it).
Need openness, censorship-resistance, no gatekeeper → public (accept ~7 tx/s and pseudonymity).

Memory hooks

Hash chain = tamper-EVIDENT, not tamper-proof.
Immutability ≠ correctness — a confirmed wrong entry is permanently wrong.
Consensus = every node applying the same rules — no vote, no committee.
Miners earn = block reward + fees; losers validate the winner.
Signed by private key at creation; final ~6 blocks later.

Smart contract red flags

Code is law including the bugs; needs outside data → oracle = new trust bottleneck; executes decisions about people → Art. 22; agent + contract + finality → irreversible autonomous mistakes (cap limits, human gate).

🧩 Case-answer rules of thumb

The 5-line pre-write (60 seconds)

CONCEPT → which pattern? · WHY → mechanism & who's harmed · CASE → which analogy from the bank? · X-DOMAIN → which of the 4 connections? · TRADE-OFF → what does my fix cost?

Then write. Steps 4–5 are the 7→12 jump.

Future-risk generator

Pick the engine that makes it worse: feedback loop (outputs → training data) · compounding (0.95¹⁰≈60%) · drift (world moves) · scale (small % × millions) · HNDL clock · finality (mistakes permanent) · regulatory exposure accruing.

Fix selector

Match fix to diagnosis: label bias → label audit · proxy bias → subgroup fairness audit · feedback loop → break the loop · representation → fix the dataset · activity proxy → verification step · output proxy → grounding + validation · self-reported → independent audit · silent failure → monitoring + owner · agentic → checkpoints + human gate + kill switch · scale-before-data → staged rollout (Waymo).

Cross-domain picker (choose 1, one sentence)

Threshold/error rates → discrimination → EU AI Act.
Anything stored/immutable → GDPR erasure collision.
Anything encrypted + long-lived → HNDL/PQC.
Anything autonomous + transacting → agentic × smart contracts accountability gap.

End with the trade-off: "this fix costs X — accepted, because Y is the disaster here."

⚠ Trap pairs — the distinctions MC questions live on

AI / ML

Overfitting vs. Distribution shift AI/ML

Overfitting = fails on new data from the same world (memorised the training set — "the chef who memorised one cookbook").

Distribution shift = the world itself changed after training (2021 fraud model meets 2023 fraud patterns).

Teacher's recap flags this as "the two most confused terms." Clue words: "performed well at first, then degraded" → shift. "Great on training/validation, bad from day one in production" → overfitting.

Data drift vs. Concept drift AI/ML

Data drift: the inputs change (new transaction types, new customer mix). Same task, different world.

Concept drift: the input→output relationship changes (what "creditworthy" means shifts after a recession).

Both are forms of distribution shift / model drift. The confidence trap: the model doesn't know it drifted — it keeps giving high-confidence answers. High confidence ≠ high accuracy.

Precision vs. Recall AI/ML

Precision = TP/(TP+FP) — "Of my calls, how many were right?" A column question (everything YOU flagged).

Recall = TP/(TP+FN) — "Of what was real, how much did I catch?" A row question (everything REALLY there).

Same TP numerator, always. The denominator tells you which question you're asking. Miss (FN) = disaster → chase recall. False alarm (FP) = the expensive one → chase precision.

NLP vs. Generative vs. Agentic AI/ML

NLP reads (text in, meaning out) · GenAI creates (prompt in, content out) · Agentic acts (goal in, action out).

Practice-test clue: a chatbot that answers is generative; the moment it initiates refunds / sends emails / updates the CRM autonomously it has become agentic. Flagging complaint emails = NLP classification.

Supervised vs. Unsupervised vs. Reinforcement AI/ML

Supervised: labelled examples (fraud/not fraud) — most business AI. Unsupervised: no labels, finds structure (segmentation, clustering). Reinforcement: trial and error with rewards/penalties (trading bots; the agentic loop "slaps itself on the wrist").

Clue: "segment 2M customers into natural groups, no predefined categories" → unsupervised.

RAG vs. Fine-tuning AI/ML

RAG: retrieve current documents at query time, ground the answer in them — cheap per query, knowledge stays current. Fine-tuning: bake knowledge/behaviour into the weights ($10K–500K) — good for style/tone, knowledge frozen at fine-tune time.

Most enterprise stacks combine both: fine-tune for behaviour, RAG for facts.

Bias vs. Variance (trade-off) AI/ML

High bias = model too simple → underfits (misses real patterns). High variance = model too complex → overfits (learns noise). You are always trading one risk for the other.

An undertrained transformer = high bias end: it hasn't learned the patterns yet — outputs are generic/random, poor even on training data.

Hallucination — cause, not bug AI/ML

LLMs pick the most statistically plausible continuation, not the most true one — they cannot tell the difference. Plausible ≠ correct is "the defining exam trap."

Consequences: no reliable arithmetic (numbers are just tokens), no current facts (training cutoff), fabricated citations. Mitigation: RAG grounding + human verification — mitigates, never eliminates.

Data Governance

Data controller vs. Data processor Governance

Controller decides why and how data is used (bears main accountability). Processor only handles data on the controller's instruction.

Ownership rule from the course: business owns (accountable for accuracy, purpose, compliance), IT manages (storage, access, security), steward monitors (day-to-day quality).

Compliance vs. Fairness Governance

Compliance: may the company legally use the data (consent, purpose, legal basis)? Fairness: are the resulting decisions equitable? You can be fully GDPR-compliant and still discriminatory (location proxies income).

Separating these two is the teacher's own tier-5 example on the governance question — say both words.

Data drift response vs. Erasure response Governance

Model got worse over time → drift monitoring + retrain (a performance problem). Person wants their data gone → Art. 17 erasure options (a rights problem: retrain / differential privacy / document + assess).

Both end in "retrain" — but the trigger and the justification differ. Name the right trigger.

Quantum

Superposition vs. Entanglement vs. Interference Quantum

Superposition: one qubit in multiple states (0 AND 1) at once, until measured. Entanglement: two qubits linked so measuring one instantly fixes the other (anti-correlated). Interference: wrong answers cancel out, right answers reinforce — the mechanism that makes quantum algorithms actually compute.

MC clue: "wrong answers cancel, correct answer amplifies before measurement" → interference (NOT superposition).

Grover vs. Shor Quantum

Grover's = search speedup (quadratic: √N checks instead of N/2). Shor's = factoring large numbers (exponential speedup) → breaks RSA and ECC/ECDSA encryption.

Anything about "breaking encryption" = Shor. Anything about "searching a huge unstructured space faster" = Grover.

HNDL — the threat is NOW Quantum

Harvest Now, Decrypt Later: adversaries store your encrypted data today and decrypt it when quantum arrives. "RSA is currently unbroken" is irrelevant for data that must stay secret 10+ years.

This is what turns quantum from a future problem into a governance decision you must make today (PQC migration).

Overfitting-hype vs. real quantum claims Quantum

A credible claim names hardware + algorithm + problem shape and admits NISQ limits. A hype claim promises speedups on "all workloads", "room temperature", "no error correction", or "next year".

Only 4 problem shapes benefit: optimisation, simulation, factoring, search.

Blockchain

Public vs. Private/permissioned chain Blockchain

Public/permissionless: anyone joins, pseudonymous, transparent, slow (~7 tx/s Bitcoin vs ~1,700 Visa). Private/permissioned: known participants, fast (1,000s tx/s), privacy controls, KYC-compatible.

Why banks use private chains: control, compliance (KYC/AML), speed, confidentiality. Practice-test Q14 is exactly this.

Tamper-evident ≠ tamper-proof · Immutability ≠ correctness Blockchain

The hash chain makes changes detectable (change one block → every hash after it breaks) — that's tamper-evidence. And a wrong entry, once confirmed, is permanently wrong.

Flagged in the recap as "a favourite short-answer trap."

Wallet vs. Node vs. Miner Blockchain

Wallet: holds your keys, builds & signs transactions (not a node). Node: validates & propagates (no single owner). Miner: a specialised node that bundles transactions into blocks and races to seal them.

Losing miners validate the winner's block — validation is separate from winning.

Consensus = rule-following, not voting Blockchain

Every node independently validates the new block against the same rules (valid txs? correct prev hash? puzzle solved?). If valid → append. No committee, no vote. The "majority agrees" emerges from every node applying identical rules.

Also: consensus keeps distributed copies identical — completing the loop with hash chain (detectable) + distributed ledger (impractical to tamper).

📐 Formulas

Accuracy

(TP + TN) / (TP + TN + FP + FN)

"How often right overall?" — misleading on imbalanced data (99.8% accuracy can mean 0 fraud caught if fraud is 0.2%). Always ask for the base rate.

Precision

TP / (TP + FP)

Column question — of everything I flagged, how much was real? Chase when false alarms are the expensive mistake.

Recall

TP / (TP + FN)

Row question — of everything real, how much did I catch? Chase when a miss is the disaster.

F1 score

F1 = 2 × (P × R) / (P + R)

Harmonic mean of precision & recall — punishes imbalance (if either is low, F1 is low). Use as a single number when both error types matter and classes are imbalanced. E.g. P=0.56, R=0.95 → F1 = 2(0.532)/1.51 ≈ 0.70.

Error compounding

end-to-end ≈ (per-step accuracy)^steps

0.95¹⁰ ≈ 0.60 · 0.9⁷ ≈ 0.48. Why agentic workflows need checkpoints and human gates.

The confusion matrix — plot it right, every time

Rows = reality (actual). Columns = the model's call (predicted). Worked example from the practice test: 1,000 patients, 200 truly need follow-up, model flags 340, of which 190 are correct:

PREDICTED: FLAG (340)
PREDICTED: CLEAR (660)
ACTUAL: NEEDS (200)
TP = 190 — caught it
FN = 10 — missed it (the disaster)
ACTUAL: FINE (800)
FP = 150 — false alarm (the annoyance)
TN = 650 — correctly quiet

Precision = 190/340 = 56% (low) · Recall = 190/200 = 95% (high) → "low precision, high recall — catches almost everything, at the cost of unnecessary referrals." That's a deliberate design choice because a missed case is worse here. Your teacher's version: "of 1,000 cases, 200 are real fraud" — same plot: real fraud = top row, model flags = left column, then read off the four cells.

AI / ML — how AI learns

What is the train/validation/test split, and why does it matter?

The data is split (course slides: 70% train / 15% validation / 15% test). The model learns on the training set, hyperparameters are tuned on the validation set, and the test set is touched once, at the end, to estimate real-world performance. Why it matters: if you evaluate on data the model has seen, you measure memorisation, not understanding — you get a flattering score that collapses in production (overfitting goes undetected).

⤴ Upgrade: "An undertrained transformer sits at the other extreme: it hasn't yet learned the underlying patterns at all, so it performs poorly even on training data (high bias/underfitting), producing generic or incoherent output. Training curves on train vs validation data tell you which side of the bias–variance trade-off you're on." + governance angle: the held-out test set is your only honest evidence for regulators.

Explain overfitting.

Overfitting is when a model learns the training data too specifically — noise and coincidences included — instead of the general pattern. The course analogy: a chef who memorises one cookbook perfectly — flawless on any dish from that book, lost with a new ingredient. He learned the book, not how to cook. Symptom: near-perfect training accuracy, poor performance on new data from the same world.

⤴ Upgrade: distinguish from distribution shift (world changed) and connect: "an overfit model is ungovernable in production — it will fail unpredictably when the world moves, so model monitoring is the governance response." Bias–variance framing: overfitting = high variance; underfitting = high bias.

Explain distribution shift and its implications.

Distribution shift is when the world changes after training. Two forms: data drift — the inputs change (2022 fraud patterns → 2026 patterns); concept drift — the relationship between inputs and outcomes changes (what "creditworthy" means after a recession). The implication: accuracy degrades silently. The model doesn't know it drifted — it keeps producing high-confidence outputs because it's still internally consistent. High confidence ≠ high accuracy.

⤴ Upgrade: "The governance response is drift monitoring: performance thresholds, automated alerts, policy-driven retraining — most organisations have none and find out when a decision fails publicly. The Lucas Critique is the extreme case: deploying the model changes the very behaviour it was trained to predict (Dutch benefits enforcement changed applicant behaviour)."

Was it the model or the data? (the 80/20 question)

Assume the data first. 80% of ML work is data collection, cleaning, labelling and validation; only 20% is modelling — and the #1 root cause of model failure is poor data quality, not algorithm choice. Checklist: were labels themselves valid (label bias)? was the data representative? has the world changed since collection (drift)? were there quality gates at ingestion? Only after ruling those out, look at model complexity (over/underfitting) and deployment (monitoring, verification).

⤴ Upgrade: "Amazon's hiring model worked exactly as designed — learning from the data it was given. That was exactly the problem: the labels ('hired/not hired') encoded a decade of bias. Its failure was decided before a line of model code was written."

Choose precision vs recall for scenario X (template)

Step 1 — define both: precision = TP/(TP+FP), of my flags how many were right; recall = TP/(TP+FN), of the real cases how many I caught. Step 2 — name the two errors in this scenario and ask: which mistake is the disaster? If the miss (FN) is the disaster → chase recall, tolerate noise (wildfire evacuation, cancer screening, data-centre overheat). If the false alarm (FP) is the expensive/dangerous one → chase precision, act only when sure (small-business spam filter — a real customer email lost is the disaster; AML when review capacity is 40 cases/day but you flag 200).

⤴ Upgrade (this exact upgrade is the teacher's tier-5 example): "The threshold is not just a modelling choice — it's a governance choice. Precision failures (wrongly denied customers) can cluster by demographic → discrimination risk → EU AI Act high-risk exposure for credit scoring. A model tuned only to the bank's loss function is optimising the wrong objective — Goodhart's Law in a credit context." Also note capacity: recall is pointless if the review team can't process the flags.

What is the F1 score and when do you use it?

F1 = 2·(P·R)/(P+R) — the harmonic mean of precision and recall. Unlike a simple average, it punishes imbalance: if either P or R is near zero, F1 is near zero. Use it as a single summary metric when both error types matter and classes are imbalanced (where plain accuracy is misleading). Example: P=56%, R=95% → F1 ≈ 70%.

⤴ Upgrade: "F1 weights precision and recall equally — but business costs rarely are equal. If FN costs 10× FP, report precision and recall separately and set the threshold from the cost asymmetry, rather than optimising F1 blindly — otherwise the metric becomes the target (Goodhart)."

Explain the transformer architecture / attention.

Before 2017, models processed text sequentially — by the time they reached the end of a sentence, early context was fading, and training couldn't parallelise. The transformer ("Attention Is All You Need", 2017) processes all words simultaneously, and attention lets every word weigh its relationship to every other word — "sharply" knows it modifies "fell" which describes "market". Parallelism made billion-parameter models trainable on GPUs. Every LLM — GPT, Claude, Gemini — is a transformer.

⤴ Upgrade: "Note what the architecture doesn't change: the model still only predicts the next most probable token. Scale bought fluency, not truth — which is why hallucination is a property, not a bug." (NB: if the teacher loosely says "a transformer" meaning the whole ML pipeline/split — answer the concept he's pointing at: the trained model and its train/val/test discipline.)

How does an LLM actually work — and what follows from that?

An LLM is trained on internet-scale text to predict the next token. At inference it generates one token at a time, each the most probable continuation given everything before, until an end-of-sequence token. No reasoning, no understanding, no database lookup — only very good prediction. Three consequences: hallucination (plausible ≠ correct — it can't tell the difference), no reliable arithmetic (numbers are tokens), no current facts (training cutoff).

⤴ Upgrade: "Mitigations map to causes: RAG grounds answers in retrieved current documents (fixes cutoff + reduces hallucination), fine-tuning bakes in domain behaviour, human review catches what both miss. Air Canada shows the liability: its chatbot generated the most plausible policy, not the real one, and the tribunal held the company responsible for everything on its website."

Generative vs Agentic AI — and why agentic risk is different.

Generative AI answers: you prompt, it produces output, a human evaluates it — single turn, stateless, no action in the world ("a brain in a jar"). Agentic AI acts: the same LLM wrapped in a Plan → Act → Observe → Adjust loop with tool access (APIs, search, systems) — multi-step, stateful, takes real actions. The risk changes category: errors compound (0.95¹⁰ ≈ 60% end-to-end), a misread goal cascades, and there's an accountability gap — who owns an autonomous agent's action?

⤴ Upgrade: "Design answer: autonomy is a dial, not a switch — place a human in the loop wherever an action is irreversible, costly, or regulated (a credit decision is legally high-risk under the EU AI Act). Cruise shows where the loop breaks: the vehicle observed a collision, braked, then ran its pullover protocol without re-observing — the Observe step failed."

What is RAG and what problem does it solve?

Retrieval-Augmented Generation: before answering, a retrieval system fetches current/authoritative documents, and the LLM synthesises an answer grounded in them. It solves the training-cutoff problem (knowledge stays current without retraining) and reduces hallucination by anchoring generation to a source of record. Cheap per query vs fine-tuning ($10K–500K) or retraining ($50–100M).

⤴ Upgrade: "Air Canada's chatbot had no retrieval layer — grounded in the actual bereavement policy it would likely have answered correctly. Most enterprise deployments combine both: fine-tune for tone/behaviour, RAG for facts. RAG mitigates but does not eliminate — a validation gate against the source is still needed."

Data Governance

Why do 87% of AI projects never reach production?

Because the failure is usually in the data and the organisation, not the algorithm: unclear ownership (nobody accountable), poor data quality (the #1 root cause), no lineage (can't audit or explain), no quality gates, and no documentation — the five warning signs. 80% of a data scientist's time goes to cleaning data. Governance is not bureaucracy; it's the infrastructure that turns the 87% into survivors.

⤴ Upgrade: "Ownership is the foundation: business owns (accountable for accuracy, purpose, compliance), IT manages (infrastructure, access), stewards monitor daily quality. Every governance failure in the Week 2 cases traces back to unclear ownership — and regulation (EU AI Act) now assumes this infrastructure already exists."

Purpose limitation (the most-tested governance concept)

Under GDPR, personal data may only be used for the specific purpose stated when it was collected. Using it for a new purpose requires a new legal basis — typically fresh consent. The exam shape: "data collected for X, used for Y." Related: consent decay — consent given for one purpose does not silently transfer to a new one; and data minimisation — collect only what the purpose requires.

⤴ Upgrade: "Cambridge Analytica is the canonical case: data collected for academic personality research, used for political micro-targeting of 87M people. Facebook's 'we asked them to delete it' was not a governance process — no audit, no verification, no notification. GDPR (drafted 2016) is the scar tissue of exactly this case. Tier-5 move: compliance ≠ fairness — even legally-consented location data can proxy for income and produce discriminatory pricing → EU AI Act territory."

Right to erasure (GDPR Art. 17) vs a trained model

A person can demand deletion of their data. For a database: delete the row — done. For a model trained on that row: the data is baked into the weights; you cannot remove one person's contribution without retraining, and retraining is expensive. Genuine options (practice-test answer): ① delete source data and retrain; ② differential privacy / partitioning to limit an individual record's influence; ③ machine unlearning (emerging, not production-ready); ④ document why full erasure is technically complex and assess the risk — plus the only scalable answer: prevention by design — verify a legal basis for every data point before training.

⤴ Upgrade: "Same collision, second domain: blockchain immutability. A hash chain structurally cannot un-write a block — the tamper-evidence that makes it trustworthy is exactly what blocks a court-ordered or GDPR deletion (collision type A: permanence vs the right to change)."

EU AI Act — high-risk classification

The EU AI Act (2024) regulates by risk tier. High-risk systems — those affecting people's lives: credit scoring, hiring, healthcare, justice — require conformity assessment before deployment: documented training-data quality and lineage, human oversight, explainability, error-rate evidence. Fines up to 7% of global revenue. It turns a modelling choice (e.g. a threshold) into a legal one.

⤴ Upgrade: "The Act essentially asks: what are your FP/FN rates, and who is harmed by them? — the confusion matrix is the raw material of AI regulation. Gap: it assumes traceable, explainable, reproducible systems; GenAI is stochastic, web-trained and unexplainable per-decision — the regulatory gap the course keeps returning to."

Explainability — why "the AI can't explain itself" is a regulatory problem

For high-risk automated decisions, GDPR Art. 22 and the EU AI Act require organisations to explain how a decision was made and keep meaningful human oversight. Traditional ML can deliver this (feature importance, SHAP, deterministic audit trails). LLMs cannot: outputs emerge from billions of parameters, the same prompt gives different answers, and no one can trace why a specific output was produced. Regulation assumes capabilities GenAI doesn't have — projects get blocked by compliance, not technology.

⤴ Upgrade: "Responses organisations try: tiered risk frameworks (GenAI gets output monitoring + human review + use-case restrictions instead of full explainability), proxy explainability tools, separate governance tracks. Dutch benefits shows the human cost of no explainability: 26,000 families with no explanation and no route to contest."

Model drift & monitoring (governance version)

A model accurate at deployment degrades silently: data drift, concept drift, and degradation as unseen edge cases accumulate. Because the model still reports high confidence, nobody notices until a decision causes visible harm. Governance response: set performance thresholds/SLAs, automated alerts, and policy-driven retraining triggers — not ad hoc. Most organisations have no drift monitoring.

⤴ Upgrade: "Connect to deployment reality: 'the model was checked at launch' is an activity proxy — verification must be continuous. Insurance-pricing practice question: inputs unchanged but input→outcome relationships shifted because a competitor changed customer behaviour → concept drift → retrain on recent data."

Who owns data inside an organisation?

Business owns — accountable for accuracy, purpose and compliance (the accountable party). IT manages — storage, access, security, infrastructure. Stewards monitor — day-to-day quality, metadata, policy enforcement. In practice ownership often sits with whoever built the system, quality dies when they leave, and regulators get no answer — which is why "who is responsible for this data?" is the first governance question.

⤴ Upgrade: "Controller vs processor (GDPR): the controller decides why/how data is used and carries the accountability; the processor handles it on instruction. And the accountability principle: you must be able to demonstrate compliance — documentation, DPIAs, audit trails — not just claim it."

Quantum

Qubit, superposition, entanglement, interference (the four-parter)

A qubit is the quantum unit of information — unlike a classical bit locked to 0 or 1, it can hold more than one state. Superposition: a qubit exists in multiple states (0 AND 1) simultaneously until measured — measurement collapses it to a definite value. Entanglement: two qubits linked into one shared state, so measuring one instantly fixes the other, no matter the distance; hardware must bring qubits close enough to interact to create the link — entanglement is what lets many qubits act as one computational system. Interference: quantum algorithms choreograph the amplitudes so wrong answers cancel out and correct answers reinforce — that is how a quantum computer actually computes an answer rather than just holding many possibilities.

⤴ Upgrade: "Interference is the exam favourite: both Grover's and Shor's algorithms rely on it. And the honest caveat scores points: qubits are fragile (decoherence — heat/vibration destroys the state), which is why hardware runs near absolute zero and we're still in the NISQ era, ~1,000 physical qubits per logical qubit."

Grover's and Shor's algorithms

Grover's: quadratic speedup for unstructured search — ~√N checks instead of N/2. Applications: database/pattern search, fraud-pattern detection. Shor's: exponential speedup for factoring large numbers — which is precisely the hard problem RSA and ECC/ECDSA encryption rely on. A large-enough fault-tolerant quantum computer running Shor's breaks most of today's internet, banking and signature encryption.

⤴ Upgrade: "Match hardware → algorithm → problem shape: if a quantum claim can't name all three specifically, it's probably hype. Only four problem shapes benefit at all: combinatorial optimisation, quantum simulation, factoring, search. For everything else classical wins — quantum won't run Excel faster."

Harvest Now, Decrypt Later — why act today?

An adversary intercepts and stores encrypted data today, then decrypts it years from now once quantum computers can run Shor's at scale. The vulnerability exists NOW even though the capability arrives LATER: anything that must stay secret longer than the quantum timeline (mortgage contracts, health records, state secrets, pension custody records) is already at risk the moment it's transmitted under RSA/ECC. "It's not broken yet" is therefore not a safety argument for long-lived data.

⤴ Upgrade: "This turns quantum into a present-day governance decision: NIST published PQC standards in Aug 2024 (ML-KEM, ML-DSA, SLH-DSA), deprecates RSA by 2030, prohibits by 2035; migration takes 5–10 years for a large organisation — so the window to start is now. Migration sequence: inventory → prioritise long-lived secrets → adopt standards → migrate. Collision type C: present security vs a future-dated break."

The future of quantum (timelines & the honest take)

We're in the NISQ era: noisy, small-scale, no fault tolerance; error correction needs ~1,000 physical qubits per logical qubit; timelines keep slipping (advantage promised "in 5 years" in 2015, 2020, and 2025). Experts diverge on Shor-scale machines: 5–20 years; ~50% of experts say by 2035; the qubit requirement to break RSA-2048 dropped from 20M (2019) to ~100K–1M (2025–26), which shortens timelines. For most businesses: watch and track, don't bet the strategy — but do start the crypto migration (that's the urgent part).

⤴ Upgrade: "JPMorgan's quantum initiative is the governance near-miss: real $100M+ investment, real research — but 'quantum advantage in 5–10 years' quietly became 'monitoring the landscape.' They measured papers published, not competitive advantage: measurable ≠ meaningful, no feedback loop to kill the narrative."

Blockchain

Explain the trust mechanism (the three layers)

Blockchain replaces a trusted intermediary with three interlocking layers. ① Hash chain: every block contains the previous block's hash — change any past transaction and every subsequent hash breaks, so tampering is detectable. ② Distributed ledger: thousands of nodes hold identical copies — no single point of failure, so tampering is impractical (you'd have to change most copies at once). ③ Consensus: every node independently validates each new block against the same rules — no vote, no committee — so all copies stay identical. Built to remove dependence on banks: no single point of failure, no trusted intermediary, and a bank run can't take your record down.

⤴ Upgrade: "Precision matters: it's tamper-EVIDENT, not tamper-proof — and immutability ≠ correctness: a wrong entry, once confirmed, is permanently wrong. That structural permanence is also the GDPR-erasure collision (type A)."

What is mining, who wins, and how do people earn money?

Miners each assemble a candidate block from the mempool (usually highest-fee transactions first) and race to find a nonce — a number that makes the block's hash start with enough zeros. Pure brute-force search: trillions of attempts per second — expensive to do, trivial to verify (proof-of-work). The first to solve it broadcasts the sealed block; the losing miners validate the winner's block and immediately start racing for the next one. Earnings: the winner collects the block reward (newly created coin — how supply enters circulation, no central bank) plus all transaction fees in the block. That reward is the economic engine that sustains the whole network.

⤴ Upgrade: "The incentive design is the insight: mining converts economic self-interest into network security — attacking the chain costs more than honestly earning from it (until someone controls 51% of the hash power)."

When does a transaction get signed / signed off?

Two moments, and it pays to name both. Signed: at creation — your wallet builds the transaction (recipient, amount, fee) and signs it with your private key, proving authorisation without revealing the key; anyone can verify with your public key. Signed off (settled/final): after broadcast → every node independently validates (signature valid? balance sufficient? no double-spend? fee ok?) → mempool → mined into a block → consensus → and after ~6 further blocks (~60 min on Bitcoin) it reaches finality — permanent, no chargeback, no reversal.

⤴ Upgrade: "Contrast with T+2: traditional settlement locks funds for ~48 hours across clearinghouses and correspondent banks with counterparty risk in the gap; blockchain settles in minutes with no exposure window. Key management is the flip side: whoever holds the private key controls the funds — no reset, no bank to call. You are the bank."

What is tokenisation?

Representing a real-world asset (bond, fund share, real estate) as a tradeable digital token on a blockchain. Live examples from the course: BlackRock's BUIDL money-market fund on Ethereum ($500M+), Franklin Templeton on Polygon. Why: near-instant settlement, fractional ownership, programmable assets, 24/7 markets. Projected $10–16T of tokenised assets by 2030 (McKinsey/BCG).

⤴ Upgrade: "The Tokenisation Dilemma scenario shows the collisions: on a public chain, beneficiary data conflicts with GDPR privacy-by-design; right to erasure is impossible on-chain; and Ethereum's ECDSA is quantum-vulnerable — a €2B bond portfolio on rails that Shor's could break, where migration needs global consensus. One decision, three domains."

Why do banks use private/permissioned chains — and when is blockchain the wrong tool?

Private chains give: known participants (KYC/AML compliance), transaction privacy from competitors, speed (1,000s tx/s vs Bitcoin's ~7), and clear governance. Examples: JPMorgan Quorum/JPM Coin ($1B+ daily), R3 Corda (DTCC, HSBC), Hyperledger. Wrong tool test (Wüst & Gervais): if the parties already trust each other — same parent company, one authority — a shared database is faster, cheaper and simpler. Blockchain's value is removing the need for trust between independent, potentially adversarial parties. (Collision type D: necessity vs overhead.)

⤴ Upgrade: "The failure pattern is governance, not technology: TradeLens, ASX CHESS, We.Trade all died because competitors wouldn't join a platform a rival controlled. What worked — JPM Coin, HSBC FX Everywhere, BUIDL — had narrow scope, clear efficiency gain, controlled participants. Practice-test Q15 is exactly this."

Cross-domain (the 10–12 zone)

★ Agentic AI × Blockchain (the teacher's favourite)

Both are systems that act without a human in the moment: an agent plans and executes autonomously; a smart contract executes automatically when conditions are met. Combine them — AI agents transacting on blockchain rails — and you get machine-to-machine finance with three compounding problems. ① Accountability gap squared: no framework exists for who owns an autonomous agent's action, and the chain's finality means the action can't be reversed — an agent's mistake is permanent (error compounding meets irreversibility). ② GDPR Art. 22: a smart contract executing automated decisions about people is automated decision-making that requires explainability and human override — neither of which the stack can provide. ③ No jurisdiction: the chain has no owner and no off switch; who do you regulate?

⤴ Upgrade: "Design mitigation: keep a human gate wherever the action is irreversible (which on-chain is: everything), cap agent transaction limits in the smart contract itself, and log agent reasoning off-chain for auditability. Timeline from the course: agentic AI on blockchain rails is a 5–10 year emerging connection with no accountability framework — governance must precede deployment, not follow it."

Quantum × Blockchain (ECDSA breaks)

Blockchain security rests on ECDSA signatures (wallets) and hash assumptions — Shor's algorithm breaks ECDSA. When quantum arrives, an attacker could forge signatures and spend from any exposed wallet. Worse, the fix is uniquely hard: a bank upgrades its own systems unilaterally, but migrating a decentralised chain to post-quantum crypto requires consensus from thousands of independent nodes, simultaneously. HNDL applies to on-chain records too — and everything on-chain is public and permanent, so harvesting is trivial.

⤴ Upgrade: "The Post-Quantum Bank scenario: 'we can fix our own systems — can we fix the shared ones?' A consortium ledger (R3 Corda with HSBC, DTCC, Barclays) needs every counterparty to agree and act at the same time. Collision type C (present security vs future break) landing on top of a coordination problem."

AI × Governance (lineage & audit)

Governance is not a constraint on AI — it is the infrastructure that makes AI deployable in regulated industries. Lineage lets you trace every output back through the pipeline (required for EU AI Act audits, GDPR explanations, Basel/BCBS 239 risk models). Without lineage you cannot explain outputs to regulators — the most likely consequence of deploying GenAI without governance infrastructure.

⤴ Upgrade: "The organisations that lead build AI on a governance foundation from day one — governance as competitive advantage, not compliance cost. The ones that struggle bolt it on after deployment: technically possible, commercially and reputationally very costly."

Governance × Quantum (PQC is a governance programme)

Post-quantum migration is not a 2035 technical event — it's an enterprise governance decision made today under uncertainty: inventory what encryption protects what data, classify by how long each secret must live, prioritise long-lived secrets, adopt NIST standards, run hybrid (classical + PQC in parallel) with a rollback path. The trade-off: certain present cost ($800M–1.2B for a large bank) vs uncertain future risk (retroactive compromise of everything harvested meanwhile).

⤴ Upgrade: "Fiduciary time horizons change the math: a pension fund's 40-year liabilities outlast every quantum timeline — 'prudent fiduciary duty' vs 'speculative spend' is a defensible judgment call either way, but doing nothing must be an explicit decision, not a default. Regulators already expect migration plans — inaction is an audit finding."

🔍 Taxonomy 1 — Three failure categories (the overall shape of the story)

1 · Metric Corruption

The organisation optimises a number, and the number has quietly stopped measuring the thing it was supposed to stand for. The system usually works perfectly — that is the trap: the KPI improves while the real goal (welfare, fairness, quality) decays out of sight.

Choose this when: a KPI, target, bonus or budget drives behaviour, and the case shows a gap between the number and reality (complaints rise while the dashboard celebrates; tickets closed fast but problems unfixed). Ask: "what does the metric NOT capture, and who lives in that blind spot?" Anchors: YouTube (watch-time ≠ welfare), Uber Sydney (surge = supply signal, always), Dutch benefits (budget conditional on "fraud found"). Usually pairs with Goodhart/Campbell and often with a proxy-creep shape.

2 · Organisational Failure

The technology is almost beside the point — the organisation's own structure, incentives or politics produced the failure, and would have produced it with any technology. The foundation was broken, and they built anyway.

Choose this when: the case features incentives that reward the wrong thing, leadership buying what it cannot evaluate, silos that don't talk, suppressed evidence, or two parties who both benefit from a failing project continuing. Ask: "if I replaced the AI with a spreadsheet, would this organisation still have failed?" If yes — it's organisational. Anchors: Post Office Horizon (institution believed the machine over 900 consistent humans), IBM Watson (both actors needed the project alive, neither needed it to work), Theranos (a board of celebrities with no power to verify).

3 · Adaptive Failure

Time is the villain: the world moved and the model stood still — or, in the sharpest version (the Lucas Critique), deploying the system itself changed the behaviour it was trained to predict, so the model invalidated its own training data.

Choose this when: the case says "worked at launch, wrong now", the training data comes from a different era, or the people being modelled react to the model (fraudsters adapt, applicants game the criteria, markets respond). Ask: "did reality change on its own (drift), or did the system's presence change it (Lucas)?" — naming which one is a precision point. Anchors: Dutch benefits (enforcement changed applicant behaviour), any drift scenario, Watson (synthetic scenarios met real patients).

Can a case be two shapes at once? Yes — say so.

The strongest cases deliberately overlap: Dutch Benefits is metric corruption (fraud-found budget) AND organisational (perverse incentive design) AND adaptive (Lucas). If two fire, name the primary shape — the one that, if fixed, would have prevented the most harm — then note the second in one sentence. "Primarily metric corruption, aggravated by an organisational incentive to keep the number rising" reads as judgment; listing three shapes with equal weight reads as hedging.

🔍 Taxonomy 2 — Four types of bias (name the mechanism, the cure differs)

⚖ How to tell the four apart — ask WHERE the problem lives

Work through this in order, and stop at the first yes. ① Is the label/outcome itself a past human decision that could be biased (hired, admitted, approved, prosecuted)? → historical/label bias. ② Was the sensitive attribute removed, yet outputs still track it because correlated input features rebuild it (postcode→ethnicity, part-time years→gender)? → proxy bias. ③ Do the model's own decisions shape what future training data can contain (rejected people generate no outcome data; recommended content generates more engagement)? → feedback loop bias. ④ None of the above — the data simply under-covers a group (few dark-skin images, one accent, one scanner type)? → representation bias. They can stack: Amazon Hiring was ① AND ②. If two apply, name both and say which is upstream — the label problem usually is.

1 · Historical / Label bias

The label itself encodes a past biased decision. Amazon: "hired/not hired" carried a decade of bias. Hospital drill: "past spending" as label for "medical need" — spending reflected access, not need. Lives in the label, not the inputs.

Cure: audit whether labels are a valid measure of the real target before training; relabel or find outcome-based ground truth.

2 · Proxy / Measurement bias

You exclude the sensitive attribute, but correlated features reconstruct it. Amazon: gendered words stood in for gender; corrected, it found new proxies. Insurer drill: dropped gender, kept occupation + vehicle type → pricing still split by gender.

Cure: test outputs for correlation with the excluded attribute (fairness audit by group), not just remove the column. Note: this is what your notes call "poor proxy / label bias" — mitigate with better proxies and better training data.

3 · Feedback loop bias

The model's own outputs become tomorrow's training data — early bias compounds instead of self-correcting. YouTube: engagement → recommendations → engagement. Lending drill: rejected Group A never generates repayment data → ever fewer Group A success stories to learn from.

Cure: break the loop — hold-out populations, explore/exploit randomisation, collect outcome data independent of the model's decisions.

4 · Representation / Sampling bias

No bad label, no proxy — a group is simply underrepresented in the training data. Google Photos: darker skin tones under-sampled; failure baked in before any code. Voice-assistant drill: trained on one accent group.

Cure: fix the dataset (representative sampling), test performance per subgroup before launch, diverse QA ("a diverse QA team could have caught this by testing on themselves").

🔍 Taxonomy 3 — Three shapes of Proxy Creep

⚖ How to tell the three apart — ask WHAT is being trusted

All three are the same disease — a stand-in accepted in place of verification — but the stand-in differs. ① Trusting that a task was completed ("deployed", "migrated", "patched", "rolled out") without checking it worked? → activity proxy. ② Trusting that an output is correct because it exists and sounds fluent (chatbot answers, generated summaries, translations)? → output proxy. ③ Trusting the claimant's own numbers ("99% accurate, validated internally", self-designed benchmarks)? → self-reported metric proxy. The tell-word in the case is usually a completed-sounding verb or a suspiciously precise self-quoted figure. Proxy creep frequently co-occurs with metric corruption — the proxy IS the corrupted metric viewed up close.

1 · Activity proxy

"We did the activity" mistaken for "we verified it worked." Knight: 8 servers updated felt like 8 verified. Starbucks: rollout = validation. Drill: "migration complete" but nobody queried the new database.

Cure: independent verification step between doing and declaring done — automated config checks, acceptance tests, staged pilots with benchmarks.

2 · Output proxy

"The system produced an answer" mistaken for "the answer is correct." Air Canada: fluent chatbot answer treated as verified policy. Drill: fluent translation shipped unchecked.

Cure: grounding + validation gate against the source of record (RAG grounded in the actual policy; human check where stakes are high). Fluency is not accuracy.

3 · Self-reported metric proxy

A party's own measurement stands in for independent validation. Pieces: self-defined "0.001% hallucination rate," never audited. Theranos: "trade secret" hid everything. Drill: vendor's own 99% benchmark, never reproduced.

Cure: third-party audit, industry-standard metrics, reproducible benchmarks, contractual right to test. Ask: who measured this, and who checked them?

🔍 Taxonomy 4 — Four property-vs-requirement collisions

When does this taxonomy fire at all? Only when an inherent, by-design property of a system (immutability, pseudonymity, encryption with an expiry assumption, decentralised trust) runs into an external requirement (GDPR, AML/KYC, court orders, long retention periods, a business that doesn't actually need the property). If the case has no blockchain, no cryptography with a long horizon, and no permanence-vs-deletion tension — skip this taxonomy entirely and say nothing. Spotting a collision correctly is high-value precisely because it's cross-domain by nature; forcing one where none exists is the classic over-reach.

Domain orientation — where each collision actually lives: A (permanence vs change) → blockchain, occasionally trained model weights vs erasure. B (anonymity vs accountability) → blockchain / crypto / CBDC, basically always — the no-identity property is cryptographically enforced, so you can't subpoena a chain into knowing who owns a wallet. C (present security vs future break) → quantum/encryption, always. D (necessity vs overhead) → blockchain, always. So in practice: no chain and no long-lived encryption in the case = no collisions = write nothing. Memory hooks: A = "can't delete it" · B = "can't say who" · C = "safe today isn't safe long enough" · D = "didn't need a chain at all".

A · Permanence vs. the right to change Blockchain

Immutability vs GDPR erasure / court-ordered correction. The tamper-evidence that makes a chain trustworthy is exactly what blocks deletion — you cannot quietly un-write a block.

Fixes: keep personal data OFF-chain with on-chain hashes only; store consent pointers not records; private chain with agreed rewrite governance if trust allows.

B · Anonymity vs. accountability Blockchain / CBDC

Public wallets need no identity; AML/tax law demands identification above thresholds.

Fix pattern from the course drill: verify identity only above the legal threshold; keep the public chain below it. Layered KYC at the on/off-ramps.

C · Present security vs. future-dated break Quantum

HNDL — safe today by an assumption with a known expiry. 40-year documents / 30-year trial data under RSA = already at risk.

Fix: classify data by secrecy lifetime; anything outliving the quantum timeline migrates to PQC first; crypto-agility so algorithms can be swapped.

D · Necessity vs. overhead Blockchain

Blockchain only pays for itself when independent, mutually-distrusting parties need shared truth. Sister companies under one parent proposing a chain → ask "would a shared database do?" (Wüst & Gervais).

Fix: if a trusted party exists, use a database. Blockchain = trust problem between strangers, not an upgrade for its own sake.

⚖ The Laws wall (recognition, with case anchors)

How to cite a law well: cite at most 2–3 per case, each with one line of case evidence, and let the law do argumentative work — "this is Goodhart's Law: the moment tickets-closed became the target, it stopped measuring service quality" — rather than name-dropping. A law without evidence scores like a guess; evidence without the law's name leaves points on the table (name the named things). If two laws overlap, pick the one whose mechanism matches, not the more famous one.

⚖ Commonly confused pairs — the quick separations

Goodhart vs Campbell vs McNamara: Goodhart = a measure made a target stops measuring (mechanism). Campbell = the more an indicator drives decisions, the more it gets corrupted — same family, corruption/gaming emphasis; citing them together as "the Goodhart/Campbell family" is safe. McNamara = a step earlier: choosing to optimise only what is countable and ignoring what isn't.

Peter vs Dunning-Kruger: Peter = structural — promotion lifts people to their level of incompetence (about hierarchies). Dunning-Kruger = psychological — the less someone knows, the more competent they feel (about individuals). The exec buying unverifiable AI = Peter; the exec overruling the experts after a weekend course = Dunning-Kruger.

Murphy vs Chesterton's Fence: Murphy = the untested safeguard fails exactly when needed (test your failsafes). Chesterton = don't remove a safeguard before understanding why it exists (respect existing fences). One is about neglect, the other about deletion.

Conway vs Ashby: Conway = the system's shape mirrors the org chart (three silos → three formats). Ashby = the controller's capacity is too small for the system's variety (5 reviewers vs 40M customers, two outputs for infinite circumstances).

Parkinson vs Brooks: Parkinson = work expands to fill time/budget (Watson: $62M, 5 years, 0 patients). Brooks = adding people to a late project makes it later (coordination overhead). Budget bloat = Parkinson; headcount backfire = Brooks.

Gall vs "scale before validation": Gall = complex systems designed from scratch never work — an ideation problem (Watson tried all of oncology at once). Scale-before-validation = the thing might work, but was rolled out before anyone checked — a deployment problem (Starbucks, Cruise, raising Astrid's ceiling).

🛠 Fix & redesign toolbox

The 7 conditions for viable AI deployment (Week 3 framework — walk through ALL seven in a redesign; name which ones the case breaks)

1 · Clear scope

The system has defined use-case boundaries: it does one well-specified thing, and everyone knows where it stops. Narrow beats grand — COIN did one document type, validated it, then extended; Watson tried "all of oncology" from scratch and never worked (Gall's Law).

Test questions: Can you state in one sentence what the system decides and what it explicitly does NOT decide? Is there a boundary case everyone would agree falls outside? Broken looks like: mission language instead of scope ("revolutionise X"), scope that grew after deployment, or a chatbot answering policy questions nobody scoped it for.

2 · Data fitness

The data actually matches what the system needs: labels that measure the true target (not a biased stand-in), representative coverage of everyone the system will meet, real rather than synthetic data, and ground truth wherever it exists (Stripe: reality labels every transaction as fraud or not).

Test questions: Who created the labels, and could those decisions have been biased? Which groups are thin in the data? Has the world changed since collection? Broken looks like: "18 years of our own decisions" as ground truth, synthetic training scenarios, or a dataset predating a structural change (COVID, a recession, a new competitor).

3 · Feedback loops

Information about real-world outcomes actually travels back to people who can act: dashboards someone owns, alert thresholds that trigger escalation, complaint channels wired to the model team — it must be someone's job to read the 97 error messages Knight Capital ignored.

Test questions: If the system started failing for one subgroup tomorrow, what signal would fire, and on whose desk? How long until anyone knew? Broken looks like: a throughput dashboard that can't see error quality, complaints rising while "nothing unusual" shows, or monitoring that exists but has no named owner.

4 · Recovery from failure

The system can be stopped, and its harms can be undone. A kill switch or circuit breaker at a defined loss level, a staged rollback path, and a hybrid/parallel run before any cutover — because some harms (a dragged pedestrian, an on-chain transfer, a wrongful prosecution) cannot be recovered from after the fact.

Test questions: What loss level triggers automatic stop? Can an affected person be made whole? Is there a manual fallback process if we switch it off? Broken looks like: no off switch (Knight), irreversible actions taken autonomously, or a legacy process already dismantled so there's nothing to roll back to.

5 · Right measurement

You measure what matters, not what is easy: precision AND recall, by subgroup, against the base rate — validated independently, never self-reported. A single flattering accuracy figure on imbalanced data is the signature of this condition failing.

Test questions: Would this metric notice the failure mode we most fear? Who verified it, and were they independent of whoever built it? Does the KPI create an incentive to game it (Goodhart)? Broken looks like: "99.1% accuracy, validated internally", cost-per-application as the only number, or "papers published" standing in for capability (JPMorgan quantum).

6 · Explainability

You can explain the decision to the affected person and to a regulator: lineage from raw data to output, a model card with a named owner, and per-decision reasoning where the EU AI Act and GDPR Art. 22 require it. For GenAI this is structurally hard — which is itself a deployment decision to justify.

Test questions: If a rejected applicant asks "why?", what do they receive? Could you reconstruct which data and model version produced one specific decision? Broken looks like: "black box, but it works", no appeal route, or explanations that are post-hoc stories rather than traces.

7 · Human backstop

A human can override the system's decision, at scale, with real authority — and the override must be genuine. The course's sharpest test: check the override rate. A reviewer who processes a file in 11 seconds and overrides 1 in 900 is a rubber stamp; the safeguard exists on paper only (automation bias, the Dutch Benefits pattern).

Test questions: What is the actual override rate? Does the reviewer have the time, information, and standing to disagree? What happens to a reviewer who overrides often? Broken looks like: "a human reviews everything" with near-zero overrides, review quotas that make real review impossible, or overriders quietly penalised for slowing throughput.

🔧 Technical remediation playbook — what exactly to do, and when

When to retrain a model (the four triggers)

Retraining should be policy-driven, never ad hoc — "we retrain when someone complains loudly enough" is itself a governance failure. The four legitimate triggers:

① Drift alert: monitored performance drops below a pre-set threshold/SLA — the primary trigger. This presumes monitoring exists (condition 3). ② Scheduled cadence: a fixed rhythm (e.g. quarterly) in fast-moving domains like fraud, where patterns adapt continuously — Stripe retrains continuously because fraudsters do. ③ World events: a known structural break (recession, pandemic, new competitor, new regulation) invalidates the training era — don't wait for the metrics to confirm what you already know. ④ Post-incident: after a failure or a discovered bias, as part of remediation — paired with fixing the data, not just rerunning on it.

Crucial detail: retrain on a recent, re-audited window of data (retraining on the same flawed data reproduces the same flawed model), validate against a fresh holdout, and redeploy gradually — shadow mode or champion–challenger (new model runs alongside the old, decisions compared before it takes over). Retraining fixes drift; it does NOT fix label bias, proxy bias, or a broken KPI — say which problem retraining actually addresses.

Retrain, recalibrate, or roll back? (pick the right tool)

Not every degradation needs a full retrain. Recalibrate the threshold when the model's ranking is still good but the world's base rate moved — cheaper and faster (the precision/recall dial). Retrain when the input→outcome relationships themselves changed (concept drift). Roll back to the previous model or manual process when harm is active and you can't wait for either — recovery first, diagnosis second. Fix the data, then retrain when the problem is bias: relabel, rebalance representation, or break the feedback loop; retraining on poisoned data just launders it.

Exam sentence: "Retraining is the right fix for drift — but this case's problem is the labels, so retraining without a label audit would reproduce the bias with fresher data."

The hallucination ladder — prevent → detect → respond

Prevent (design time): ground the model with RAG against the source of record (policies, fee schedules, contracts); keep temperature low for factual tasks; constrain the output format; fine-tune for domain behaviour; and — the strongest prevention — restrict the use case: don't let a generative model answer policy, legal, or numeric questions at all if a lookup can.

Detect (run time): a validation gate that checks outputs against the source before the customer sees them; output monitoring — live sampling of responses (the new GenAI governance requirement); flag low-confidence or out-of-scope answers to a human review queue; log everything so errors are findable.

Respond (after it happens): correct and disclose to affected customers (the Air Canada lesson: you are liable for what it said, and "the chatbot did it" is not a defence); feed the failure into the validation gate as a new test case; and if hallucinations recur in a category, remove that category from the bot's scope.

The honest framing that scores: hallucination cannot be eliminated — it is how next-token prediction works. The design goal is to make it rare (grounding), caught (gates), and cheap (scope + disclosure), not impossible. Any answer claiming a fix "prevents hallucination entirely" loses credibility.

Deployment discipline (the anti-Knight/Starbucks/Cruise kit)

Before launch: define success metrics and kill criteria in writing; verify the deployment itself (config checks across all servers — "deployed" ≠ "verified"); test against real-world variation, not the demo environment. Launch: pilot in a limited scope → measure against the pre-agreed benchmarks → only then scale (Waymo: 200M miles before scaling; the opposite of Starbucks' 11,000 stores at once). Run shadow mode first where stakes are high: the system decides, a human executes, disagreements are studied. After launch: monitoring with a named owner, aggregation of small distributed errors into a visible signal, and a circuit breaker at a defined loss level.

One-line summary for a redesign answer: "pilot → measure → scale, with kill criteria written down before anyone is emotionally invested in the rollout."

Monitoring design — what to actually put on the dashboard

A dashboard that can't see the failure mode is decoration. Minimum set: precision AND recall by subgroup (aggregate accuracy hides exactly the harms that matter legally); input drift statistics (is today's data still shaped like the training data?); override rate of human reviewers (≈0% = automation bias; suddenly high = model degrading); complaint/appeal rates by segment wired from the frontline to the model team; and outcome tracking where ground truth arrives later (did approved loans repay?). Every metric needs a threshold, and every threshold needs a named owner and an escalation path.

Anchor: Starbucks had no mechanism to aggregate thousands of small errors into a signal; Astrid's dashboard tracked cost-per-application and was structurally blind to postcode clustering.

🏛 Regulatory radar — what to check in every case, and what it obliges

EU AI Act — the one to check first

Structure: risk-tiered. Unacceptable-risk systems are banned (social scoring, manipulative systems); high-risk covers systems deciding over people's lives and livelihoods — credit scoring, hiring, healthcare/triage, education, justice and enforcement; limited-risk mainly means transparency duties (tell people they're talking to an AI); minimal risk is unregulated.

What high-risk actually obliges — the checklist for a case answer: a conformity assessment before deployment (not after); documented training-data quality and lineage; logging and traceability; human oversight that is genuine (a rubber stamp fails the requirement); accuracy and robustness evidence — error rates, and who bears them; and registration/documentation duties. Fines up to 7% of global revenue.

How to use it in an answer: ① say why the system is high-risk (it decides X about people); ② name the 2–3 obligations the case's system visibly violates (no lineage, performative oversight, no pre-deployment assessment); ③ the punchline — the Act "essentially asks: what are your error rates, and who is harmed by them?" — which connects the confusion matrix to the law in one sentence.

GDPR — the five-article scan

Run any case with personal data past these five: Legal basis & purpose limitation — was the data collected for this use? A new purpose needs a new legal basis (consent decay). Data minimisation — is more collected than the purpose needs? Art. 17 erasure — can they actually delete it, from databases AND models AND chains? Know the five response options. Art. 22 — is a consequential decision automated? Then people are owed an explanation, human involvement, and a way to contest. Privacy by design + DPIA — were protections architectural, and was a Data Protection Impact Assessment done before high-risk processing began?

Aggravating factors to spot: special-category data (health, ethnicity, religion — higher bar); children; pseudonymised-but-re-identifiable data (pseudonymised ≠ anonymised — GDPR still applies, the VitaLedger lesson); cross-border transfer. Applies to any EU resident's data, processed anywhere.

Liability & enforcement precedents (cite these as "this is already law")

Air Canada (2024): a company is liable for everything its chatbot tells customers — "the chatbot is a separate entity" was rejected outright. Every customer-facing GenAI deployment inherits this. Pieces Technologies (2024, Texas AG): misrepresenting AI accuracy metrics is an enforceable violation, not just marketing puffery — self-reported metrics now carry legal risk. Cruise (2023–24): the cover-up cost more than the failure — editing evidence for regulators converted an accident into a deferred-prosecution agreement and a nationwide shutdown. Dutch Benefits / GDPR: the enforcement counterfactual — pre-GDPR fines were capped trivially (£500K for Cambridge Analytica); post-GDPR the same conduct prices at ~4% of global revenue.

Use these to make future-risk concrete: "under the Air Canada precedent, every one of Astrid's chat replies is a binding commitment the bank hasn't reviewed."

Sector & specialty rules — the quick match

Bank risk models → Basel IV / BCBS 239: evidence of data quality and lineage in all risk models — the banking regulator's version of the same demands. Crypto / tokenised assets → MiCA (EU markets in crypto-assets): issuer registration and disclosure — plus AML/KYC duties above thresholds (collision B territory). Encryption & long-lived data → NIST PQC: deprecate RSA by 2030, prohibit by 2035; NSA CNSA 2.0 requires quantum-safe national-security systems by 2027 — "compliant today" has an expiry date. Health data → special-category GDPR + national health-record retention laws (the 30-year retention that collides with HNDL).

Routing rule: decisions about people → EU AI Act + Art. 22 · reuse of personal data → GDPR · money/assets on chains → MiCA + AML · bank models → BCBS 239 · anything encrypted that must outlive a decade → NIST clock. Naming the right regulator, not just "regulation", is a precision point.

Practice test (all 15, condensed — "these scenarios will not appear in the main exam")

Q1 · Six months in, team still cleaning data, no model code. Behind schedule?

No — normal. ML is 80% data work, 20% modelling; most projects fail at problem definition and data collection/cleaning, not algorithm choice.

Q2 · 2021 fraud model quietly flags less by 2023 while fraud rises; no bugs found.

Distribution shift — fraud patterns changed after training; the world moved, the model stood still. (Not overfitting: it worked fine at first.)

Q3 · 1,000 patients, 200 need follow-up; model flags 340, 190 correct. Behaviour?

Low precision (190/340 = 56%), high recall (190/200 = 95%) — catches almost all real cases at the cost of unnecessary referrals; a deliberate trade-off when a miss is worse.

Q4 · Vendor: "It reads the portfolio and reasons through options." What's misleading?

LLMs predict the most probable next output — they don't reason or understand as implied. Plausible ≠ correct, and the model can't know when it's wrong.

Q5 · Assistant now auto-initiates refunds, sends emails, updates CRM. What shifted?

Generative → agentic. From producing output to taking autonomous multi-step action in live systems (plan-act-observe loop).

Q6 · LLM for regulatory reports — genuine strengths? (multi)

Drafting structured text + summarising long documents. NOT: guaranteeing figures, verifying regulations are current — humans must check every specific claim.

Q7 · Share of AI projects abandoned before production?

70–90% (course figure: 87%, Gartner). Cause: data quality, ownership, governance — not algorithms.

Q8 · "Which system generated this field, how was it transformed, was it modified?"

Data lineage — the full journey from source to use; prerequisite for audits and regulator queries.

Q9 · Pricing model: inputs unchanged, input→claim relationships shifted (new competitor changed behaviour).

Concept drift / the world changed → update/retrain on recent data; monitor continuously. (Also a whiff of Lucas: the market adapted.)

Q10 · GDPR erasure request; customer's data trained a live credit model. Options? (multi)

Delete source + retrain · differential privacy/partitioning to limit individual influence · document complexity and assess risk. NOT: shut down all models (disproportionate).

Q11 · "Wrong answers cancel, right answer reinforces, so measurement reads the right answer." Concept?

Interference — the mechanism that makes quantum algorithms useful; Grover's and Shor's both rely on it.

Q12 · Agency stores encrypted traffic it can't read; advisor warns about 25-year mortgages. Threat?

Harvest Now, Decrypt Later — data with a 10+ year secrecy lifetime is already at risk today.

Q13 · Startup claims room-temp, million-qubit, zero-error-correction quantum computer. Sceptical because…? (multi)

All four reasons: hardware needs ~absolute zero; million error-free logical qubits don't exist; error correction needs many physical per logical qubit; room-temp operation would be a major unsolved breakthrough.

Q14 · Bank wants internal transfers: thousands tx/min, confidential, KYC. Which chain?

Private/permissioned — known participants (KYC), privacy, fast consensus. Public chains are pseudonymous and too slow.

Q15 · 3 shipping firms build shared chain; biggest hosts it; other two quit; platform dies. Why?

Governance, not technology — competitors wouldn't trust a platform controlled by their largest rival (the TradeLens pattern). Neutral governance is essential for consortia.

Name-the-law drills (W3 opening recap)

1 · Tickets-closed-per-hour rises; issues unfixed; complaints quietly rise.

Goodhart's / Campbell's family — the metric became the target.

2 · Migration given 8 weeks takes exactly 8 weeks (elsewhere done in 2 under pressure).

Parkinson's Law — work expands to fill the time available.

3 · Failover never tested in 3 years; primary dies; failover doesn't trigger.

Murphy's Law — the untested safeguard fails when needed.

4 · Rollout 3 months late; +15 engineers; six weeks later it's 5 months late.

Brooks's Law — coordination overhead outweighs added capacity.

5 · Fraud reporting in 3 inconsistent formats — one per regional office.

Conway's Law — the system mirrors the org's communication structure.

6 · 2015: "transforms the industry in 2 years." 2017: "failure." 2025: quietly underpins everything.

Amara's Law — overestimate short-run, underestimate long-run.

7 · 5-person central compliance team reviews every transaction for 40M customers; errors rise.

Ashby's Law — the controller isn't complex enough for the system's variety.

Name-the-bias drills

1 · Hospital predicts "need for care" using past healthcare spending as the label; low-income patients spent less due to access.

Historical/label bias — lives in the label: spending stood in for need but measured access.

2 · Insurer removes gender; keeps occupation + vehicle type; pricing still splits by gender.

Proxy/measurement bias — correlated features reconstruct the excluded variable.

3 · Lending model rejects Group A more → no repayment data for them → next retrain has fewer Group A successes.

Feedback loop bias — the model's own outputs shrink what future training data can show (same shape as YouTube).

4 · Voice assistant trained mostly on one accent performs worse on others; nothing labelled accent.

Representation/sampling bias — no bad label, no proxy; the group is simply under-sampled.

5 · Résumé screener trained on 10 years of "who got promoted" favours one demographic even with names removed.

Historical/label bias — the promotion outcome itself encodes past decisions. (If it also reconstructs demographics from hobbies/schools → proxy bias on top.)

6 · Imaging model worse on scans from one equipment type — far fewer such scans in training.

Representation bias — an underrepresented slice of training data.

Name-the-proxy drills

1 · Patch rolled out to every server on the list; ticket closed; nobody confirmed it's running.

Activity proxy — doing the task treated as proof it worked.

2 · Fluent AI translation published; no target-language speaker checked it.

Output proxy — fluency treated as accuracy, no check against source meaning.

3 · Startup's own unpublished benchmark shows it beats competitors.

Self-reported metric proxy — self-designed, self-graded, unreproducible.

Name-the-collision drills

1 · Land registry on a public chain; court orders a fraudulent transfer reversed; registry can't edit.

A · Permanence vs right to change — the immutability that creates trust blocks the correction.

2 · Anonymous art marketplace; tax authority requires reporting gains above a threshold.

B · Anonymity vs accountability — fix: identity verification only above the legal threshold.

3 · Pharma encrypts 30-year trial data with today's RSA, "secure for the retention period."

C · Present security vs future-dated break — HNDL: unbroken today says nothing about 30 years.

4 · Two departments sharing one database propose a blockchain "for better trust."

D · Necessity vs overhead — they already trust each other; a shared database does the job.

Annoyance-or-disaster drills (threshold calls)

1 · Wildfire evacuation alerts.

Chase recall — missed fire (FN) risks lives; unneeded evacuation (FP) is survivable.

2 · Small-business spam filter (every email could be a customer).

Chase precision — a real customer email lost (FP) is the disaster; missed spam (FN) is annoying. Note the flip: never "always recall" — find the disaster every time.

3 · Data-centre overheat auto-shutdown.

Chase recall — missed overheat (FN) = fire/destroyed hardware; unneeded shutdown (FP) = downtime.

4 · Hospital follow-up screening (the Q3 numbers).

5 · AML flags: 200/day, review team capacity 40/day.

Calibrate your answer tier (from the Exam Approach page)

What does a tier-2 (grade 0–4) answer look like?

Right words, wrong meanings: "Recall is when it remembers things." Guessing definitions from the word itself. Fix: learn what terms actually mean.

What does a tier-3 (grade 4–7) answer look like?

Correct definitions, correct conclusion, answers the question — but no why, no example, no "it depends." Fix: add the reasoning (cost asymmetry, mechanism).

What does a tier-4 (grade 7–10) answer look like?

Adds why + a concrete case + "it depends" with context (AML capacity example). Shows the trade-off, not just the conclusion.

What does a tier-5 (grade 10–12) answer look like?

Connects a second domain by reasoning: threshold choice → demographic clustering → EU AI Act high-risk → Goodhart. Or: purpose limitation → but compliance ≠ fairness → location proxies income. Same pattern, two places, connected.

The 60-second pre-writing drill for every big question
